# Await

The `<Await>` component creates an asynchronous boundary in your application. It coordinates loading states across component subtrees, making it easy to display fallback UI while data loads from an API or performs other async work.

Unlike some frameworks that use error-throwing to trigger boundaries, Retend uses **Async Cells** paired with control flow components for a cleaner, more explicit approach.

## Understanding Async Cells

Before diving into `<Await>`, you need to know about Async Cells. They're created using `Cell.derivedAsync` and let you handle asynchronous operations reactively:

```tsx
import { Cell } from 'retend';

// An AsyncCell that fetches user data
const userData = Cell.derivedAsync(async (get) => {
  const response = await fetch('/api/user');
  return response.json();
});
```

When you pass an Async Cell to a control flow component like `If`, Retend automatically registers it with the nearest parent `<Await>` boundary.

## How Await Works

Here's the flow:

1. An `<Await>` boundary appears somewhere in your component tree
2. Components inside it use Async Cells with control flow components
3. Retend detects all pending async dependencies automatically
4. The boundary renders `fallback` content while any async operation is pending
5. Once all async operations complete, the actual content appears

## Basic Usage

```tsx
import { Await, If, Cell } from 'retend';

// Async Cell for user data
const userData = Cell.derivedAsync(async () => {
  const response = await fetch('https://api.example.com/user');
  return response.json();
});

function UserProfile() {
  return (
    <div class="user-profile">
      {/* If userData is pending, the Await boundary shows fallback */}
      {If(userData, {
        true: (user) => (
          <div>
            <h2>{user.name}</h2>
            <p>{user.email}</p>
          </div>
        ),
      })}
    </div>
  );
}

export function App() {
  return (
    <main>
      <h1>Dashboard</h1>
      {/* Displays "Loading..." until userData resolves */}
      <Await fallback={<p>Loading user data...</p>}>
        <UserProfile />
      </Await>
    </main>
  );
}
```

When `userData` is still loading, you see "Loading user data...". Once the fetch completes, the user's name and email appear.

## Async Cells vs Regular Cells

This distinction matters. A regular `Cell` can hold a Promise, but it doesn't trigger `<Await>` boundaries:

```tsx
// This WON'T trigger an Await boundary
const promise = Cell.source(fetch('/api/data'));

// This WILL trigger an Await boundary
const data = Cell.derivedAsync(async () => {
  const response = await fetch('/api/data');
  return response.json();
});
```

If you want a boundary to wait for something, wrap it in `Cell.derivedAsync`.

## Multiple Async Operations

An `<Await>` boundary waits for _all_ async operations within it to complete. If you have multiple async cells in the same boundary, the fallback stays visible until everything resolves:

```tsx
import { Await, If, For, Cell } from 'retend';

const userData = Cell.derivedAsync(async () => {
  const res = await fetch('/api/user');
  return res.json();
});

const posts = Cell.derivedAsync(async () => {
  const res = await fetch('/api/posts');
  return res.json();
});

export function Dashboard() {
  return (
    <Await fallback={<p>Loading everything...</p>}>
      <div>
        {If(userData, {
          true: (user) => <h2>{user.name}'s Dashboard</h2>,
        })}

        {For(posts, (post) => (
          <article>
            <h3>{post.title}</h3>
            <p>{post.content}</p>
          </article>
        ))}
      </div>
    </Await>
  );
}
```

Both `userData` and `posts` must finish loading before the dashboard content appears.

## Manual Registration with useAwait

If you're building custom components that handle async logic yourself, you can manually register dependencies:

```tsx
import { useAwait } from 'retend';

function CustomAsyncComponent() {
  const awaitContext = useAwait();

  if (awaitContext) {
    // Register an async cell with the nearest boundary
    awaitContext.waitUntil(myAsyncCell);
  }

  return <div>Custom content</div>;
}
```

This is rarely needed, but it's available for advanced cases.

## Server-Side Rendering

`<Await>` boundaries are essential for server-side rendering. The server needs to wait for all async operations to complete before serializing HTML. Use `waitForAsyncBoundaries()` in your SSR pipeline:

```tsx
import { waitForAsyncBoundaries } from 'retend';
import { renderToString } from 'retend-server';

// Initial render triggers all async dependencies
const dom = renderer.render(<App />);

// Wait for all boundaries to resolve
await waitForAsyncBoundaries();

// Now serialize to HTML (no loading states)
const html = renderToString(dom, window);
```

---

Async boundaries are how Retend handles the common pattern of "show loading state until data arrives". Combined with Async Cells, they keep your component tree simple and your data flows explicit.
