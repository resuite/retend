# Setup and Lifecycle

Because Retend components run exactly once to build the initial DOM, the component lifecycle is incredibly simple. You don't have to synchronize your code across complex "update" phases.

Components in Retend essentially have only two lifecycle events you need to care about: when they are **created**, and when they are **destroyed**.

---

## The Two Hooks: `onSetup` and `onConnected`

Retend provides two primary functions for running side operations outside of the standard rendering path: `onSetup` and `onConnected`. In both cases, the function you pass in runs when the component is initialized, and if you return a function, that function acts as an **unmount cleanup**.

To understand which one to use, it's crucial to understand the order in which they execute.

### The Lifecycle Timeline

When a component is mounted to the screen, the hooks run in this specific order:

1.  **`onConnected` runs**: Evaluates as soon as the specific DOM element you are tracking physically attaches to the document body.
2.  **`onSetup` runs**: Evaluates logically after the initial component tree render completes.

When the component is later removed from the screen (for example, if an `<If>` condition becomes false), their respective cleanup functions run in the exact reverse order:

1.  **`onSetup` Cleanup runs**
2.  **`onConnected` Cleanup runs**

---

## `onSetup`

`onSetup` is your go-to hook for logical component initialization. It is perfect for tasks like creating global event listeners on the `window`, establishing WebSocket connections, or starting timers.

It runs exactly once when the component instance is set up.

```tsx
import { onSetup, Cell } from 'retend';

function LiveClock() {
  const time = Cell.source(new Date());

  // 1. Create a derived Cell to format the time smoothly
  const timeString = Cell.derived(() => time.get().toLocaleTimeString());

  // 2. Lifecycle hooks are typically placed just before the return statement
  onSetup(() => {
    // Initialization
    const timerId = setInterval(() => time.set(new Date()), 1000);

    // Cleanup Function
    return () => {
      clearInterval(timerId);
    };
  });

  // Because 'timeString' is a Cell, we can pass it directly into the JSX
  return <p>Current time is: {timeString}</p>;
}
```

### Idiomatic Component Organization

Unlike frameworks that re-run component functions, Retend components are essentially just setup scripts that run once. Because of this, you are free to organize your code however you like.

However, as a best practice, you should generally order your component functions in the following way:

1. **State:** Declare your `Cell.source()` containers at the top.
2. **Derivations:** Declare your `Cell.derived()` computations next.
3. **Handlers:** Define your event handler functions (like `handleClick`).
4. **Lifecycles:** Place your `onSetup` and `onConnected` hooks near the very bottom.
5. **Return:** End the function by returning your JSX.

Keeping lifecycle hooks at the bottom right before the return statement ensures that all your Cells and handler functions have already been defined and are ready to be used by those background tasks or DOM initializers.

### The Importance of Cleanup

Always remember to return a cleanup function from `onSetup` if your initialization logic runs ongoing background tasks (like `setInterval` or `addEventListener`). If you neglect this, those background tasks will continue running even after the component is completely removed from the screen, leading to memory leaks and erratic performance.

---

## `onConnected`

While `onSetup` is for logical setup, `onConnected` is for **DOM-specific** setup.

You use `onConnected` when your logic explicitly requires a physical DOM node to exist on the screen. It is ideal for measuring dimensions (like `getBoundingClientRect()`), drawing to `<canvas>` contexts, or integrating third-party imperative UI libraries (like map or chart widgets).

To use `onConnected`, you must first create a `ref` using `Cell.source(null)` and bind it to your JSX element.

```tsx
import { onConnected, Cell } from 'retend';

function CanvasDrawer() {
  // 1. Create a ref to hold our canvas element
  const canvasRef = Cell.source<HTMLCanvasElement | null>(null);

  // 2. Track connection of that specific element
  onConnected(canvasRef, (canvas) => {
    const context = canvas.getContext('2d');
    context.fillStyle = 'blue';
    context.fillRect(10, 10, 150, 100);

    // Optional cleanup when the canvas disconnects
    return () => {
      console.log('Canvas removed');
    };
  });

  return (
    <div>
      {/* 3. Bind the ref to the element */}
      <canvas ref={canvasRef} width="200" height="200"></canvas>
    </div>
  );
}
```

### Why not use `onSetup` for this?

If you tried to read `canvasRef.get()` inside `onSetup`, the physical node might not be fully mounted into the live Document Object Model yet, which means measurements and certain API calls would fail. `onConnected` guarantees the node is physically available.
