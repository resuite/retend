# Lifecycle Hooks

Since Retend components run exactly once to create the initial output, the lifecycle model is refreshingly simple. There's no complex re-render cycle to manage, no synchronization headaches across multiple lifecycle phases. Just setup and cleanup.

In Retend, you really only need to think about two moments: when a component starts, and when it ends.

## `onSetup` and `onConnected`

Retend gives you two hooks for managing side effects and initialization logic: `onSetup` and `onConnected`. Both run when the component initializes, and if you return a cleanup function from either, it runs when the component is destroyed.

The key difference is _when_ they run relative to each other:

**The Timeline:**

1. `onConnected` runs first — the moment the physical node attaches to the environment (e.g., the DOM)
2. `onSetup` runs second — after the entire component tree is initialized

When the component unmounts, cleanup runs in reverse:

1. `onSetup` cleanup runs
2. `onConnected` cleanup runs

## Setting Up Logical State

Use `onSetup` for anything that doesn't need a physical DOM node. This is where you establish WebSocket connections, create timers, set up global event listeners, or initialize any background processes.

```tsx
import { onSetup, Cell } from 'retend';

function LiveClock() {
  const time = Cell.source(new Date());
  const timeString = Cell.derived(() => time.get().toLocaleTimeString());

  onSetup(() => {
    // Start updating the time every second
    const timerId = setInterval(() => time.set(new Date()), 1000);

    // Clean up when the component unmounts
    return () => clearInterval(timerId);
  });

  return <p>Current time: {timeString}</p>;
}
```

The cleanup function is critical. If you're running background tasks like intervals or event listeners, returning a cleanup function prevents memory leaks and ensures those tasks stop when the component is removed from the screen.

## Working with Physical Nodes

Sometimes you need to interact with the actual DOM element — measure it, draw on a canvas, initialize a third-party widget. That's where `onConnected` comes in.

`onConnected` takes a ref (created with `Cell.source(null)`) and a callback that receives the connected node. This guarantees the node exists in the renderer before your code runs.

```tsx
import { onConnected, Cell } from 'retend';

function CanvasDrawer() {
  const canvasRef = Cell.source<HTMLCanvasElement | null>(null);

  onConnected(canvasRef, (canvas) => {
    const context = canvas.getContext('2d');
    context.fillStyle = 'blue';
    context.fillRect(10, 10, 150, 100);

    return () => {
      // Cleanup if needed
      console.log('Canvas is being removed');
    };
  });

  return <canvas ref={canvasRef} width="200" height="200"></canvas>;
}
```

If you tried to read `canvasRef.get()` inside `onSetup`, the node might not be mounted yet. `onConnected` removes that uncertainty — the node is guaranteed to exist.

## Structuring Your Components

Since Retend components are essentially one-time setup scripts, you have freedom in how you organize them. But a sensible pattern helps readability:

1. **State** — Define your `Cell.source()` values at the top
2. **Derived** — Add `Cell.derived()` computations next
3. **Handlers** — Write your event handler functions
4. **Lifecycle** — Place `onSetup` and `onConnected` near the bottom
5. **Return** — End with your JSX

This ordering ensures all your Cells and handlers are available when lifecycle hooks run.

```tsx
function TodoItem({ item }) {
  // State first
  const isEditing = Cell.source(false);

  // Derived state
  const displayText = Cell.derived(() => {
    return isEditing.get() ? 'Editing...' : item.title;
  });

  // Handlers
  const toggleEdit = () => isEditing.set(!isEditing.get());

  // Lifecycle last
  onSetup(() => {
    console.log('TodoItem created:', item.id);
    return () => console.log('TodoItem destroyed:', item.id);
  });

  // JSX return
  return (
    <div>
      <span>{displayText}</span>
      <button onClick={toggleEdit}>Edit</button>
    </div>
  );
}
```

The simple rule: always include cleanup functions when you set up anything that runs in the background. Your future self (and your users' browsers) will thank you.
