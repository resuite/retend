# Teleport

Some UI elements don't play well with the normal DOM hierarchy. A modal needs to float above everything. A dropdown menu needs to escape its parent's `overflow: hidden`. A tooltip needs to break free from complex z-index stacking contexts. These components need to render _somewhere else_ in the DOM—physically detached from where they're logically defined.

Retend's `<Teleport>` component lets you do exactly that: define a component in one place in your code, but render its output somewhere completely different in the actual DOM.

## Using Teleport

Wrap the content you want to move and tell it where to go with the `to` prop:

```tsx
import { Cell, If } from 'retend';
import { Teleport } from 'retend-web';

function Modal(props) {
  const { isOpen, onClose } = props;

  return If(isOpen, {
    true: () => (
      <Teleport to="#modal-root">
        <dialog class="modal-overlay" open onClick={onClose}>
          <article class="modal-content" onClick--stop>
            <h2>Modal Title</h2>
            <button onClick={onClose}>Close</button>
          </article>
        </dialog>
      </Teleport>
    ),
  });
}
```

In your HTML, set up a container for teleported content:

```html
<body>
  <div id="app"></div>
  <div id="modal-root"></div>
</body>
```

When `isOpen` is true, the modal renders inside `#modal-root`—not inside the `<Modal>` component's normal DOM position. When `isOpen` becomes false, Retend cleanly removes the modal from the target.

### CSS Selectors

The `to` prop accepts any CSS selector. Retend finds the first matching element and renders there:

```tsx
<Teleport to="#popover-container">...</Teleport>
<Teleport to="body">...</Teleport>
<Teleport to=".overlay-target">...</Teleport>
```

If Retend can't find a matching element, it logs an error to the console.

## Physical vs. Logical

`<Teleport>` only changes where elements _physically_ appear in the DOM. Logically, they're still children of the component that rendered them.

This means:

- **Reactivity works perfectly.** Cells used inside the Teleport update seamlessly.
- **Context still flows.** Scopes and Context you access from parent components work inside the Teleport.

```tsx
import { useScopeContext, Cell } from 'retend';
import { Teleport } from 'retend-web';
import { ThemeScope } from './scopes';

function ThemedTooltip(props) {
  const { text } = props;

  // Access the theme from our logical parent, even though
  // we physically render in the body
  const theme = useScopeContext(ThemeScope);

  const themeClass = Cell.derived(() => `tooltip tooltip-${theme.get()}`);

  return (
    <Teleport to="body">
      <div class={themeClass}>{text}</div>
    </Teleport>
  );
}
```

The tooltip logically belongs to its parent component's tree, so it can read any Context from above. But physically, it lives in the `<body>`, free from layout constraints.

## A Practical Example

Here's a complete modal system with overlay dismissal:

```tsx
function ConfirmDialog(props) {
  const { title, message, onConfirm, onCancel } = props;

  return (
    <Teleport to="#dialog-root">
      <div class="dialog-overlay" onClick={onCancel}>
        <div class="dialog-box" onClick--stop>
          <h2>{title}</h2>
          <p>{message}</p>
          <div class="dialog-actions">
            <button onClick={onCancel}>Cancel</button>
            <button onClick={onConfirm} class="btn-primary">
              Confirm
            </button>
          </div>
        </div>
      </div>
    </Teleport>
  );
}
```

The overlay click closes the dialog, but clicking inside the box doesn't (thanks to `onClick--stop`). The logical parent can still pass down Cells and Context. The physical rendering location keeps it above everything else on the page.

---

Teleport bridges the gap between logical component hierarchy and physical DOM structure. Use it for overlays, dropdowns, tooltips, and any UI that needs to break free from normal layout constraints.
